# ⚠️ 最重要的核心原則（所有專案必須嚴格遵守）

## 🎯 模組化三大鐵律

### 1. 所有程式都必須照 PROJECT_RULES.md 規範進行模組化
- **一個功能一支程式**：禁止多功能耦合在單一檔案內
- **單一職責原則**：每個模組只負責一件事情，職責清晰明確
- **500行上限**：單支程式碼不得超過 500 行，超過必須拆分為 core 與 utils
- **新功能新檔案**：新增功能必須建立新程式檔案，不得堆疊於舊檔案中

### 2. 避免改 A 壞 B（絕對禁止連鎖破壞）
- **範圍控制**：僅能修改明確指定的檔案與模組
- **隔離原則**：禁止「改 A 壞 B」的連鎖反應
- **影響評估**：修改前必須分析相依性影響範圍
- **回歸測試**：每次修改後必須執行相關測試，確保既有功能不受影響

### 3. 每個模組職責單一（Single Responsibility Principle）
- **功能分離**：一個模組只做一件事情
- **高內聚低耦合**：模組內部高度相關，模組之間低度依賴
- **明確介面**：模組對外提供清晰的介面定義
- **不可破壞既有功能**：絕對禁止修改或刪除既有的公開介面

## 5. 資料與狀態管理 (依照共同規範)

### 分層管理
- **全域層級**：跨模組共享、影響整體系統（使用者資訊、系統設定、快取）
- **模組層級**：單一功能模組內部使用（業務邏輯狀態、暫存資料）
- **本地層級**：單一元件/函式內部使用（臨時變數、局部計算）

### 持久化儲存原則
所有具備「紀錄」性質的資料，**必須**透過後端 API 持久化到資料庫，嚴禁僅存放於前端：

- **強制後端持久化的資料類型**：
  - 操作紀錄 / 操作日誌（誰在什麼時候做了什麼）
  - 聊天歷史 / 對話紀錄
  - 表單填寫進度 / 草稿
  - 使用者偏好設定（排序方式、篩選條件、版面配置等）
  - 通知已讀狀態
  - 搜尋歷史
  - 上傳紀錄 / 處理進度
  - 任何需要跨裝置、跨瀏覽器存取的狀態資料
- **前端定位**：前端僅作為「顯示層」，不得將紀錄類資料僅存放於前端狀態（Zustand、React State、Redux 等）或瀏覽器儲存（localStorage / sessionStorage）
- **快取允許但來源必須是後端**：前端可對後端資料做快取加速，但資料來源必須是後端 API，頁面刷新後須能從後端重新取得完整資料
- **判斷標準**：若該資料「使用者刷新頁面後消失會造成困擾」，則該資料必須後端持久化
- **API 設計**：紀錄類資料須提供對應的 CRUD API（至少包含建立與查詢），並遵循第14章 API 接口契約規範
- **資料庫設計**：紀錄類資料表須包含 `created_at`、`updated_at` 時間戳欄位，便於追蹤與排序

### 決策要求
每次涉及資料/狀態管理，必須提供分層決策說明：
- **資料名稱**與用途
- **分層級別**（全域/模組/本地）
- **生命週期**與清理機制
- **存取範圍**與權限控制
- **決策理由**與影響評估

## 7. 開發工作流程規範 (依照共同規範)

### 需求確認流程
- **需求接收**：收到任何開發需求時，必須進行需求澄清與分析
- **反問確認**：針對模糊或不明確的需求進行反問，確保理解正確
  - 功能範圍：明確功能邊界與限制
  - 技術要求：確認性能、相容性等技術規範
  - 優先級別：確認需求的緊急程度與重要性
  - 驗收標準：明確完成的定義與驗收條件
- **需求分解**：將複雜需求拆分成具體的可執行任務
- **代辦清單**：使用 TodoWrite 工具建立並維護任務清單
- **確認完成**：需求方確認需求理解正確後才開始開發

### TodoList 管理規範
- **強制使用**：所有複雜任務（預計超過3個步驟）必須建立 TodoList
- **任務分解**：大任務必須拆分成小於500行程式碼的子任務
- **即時更新**：任務狀態變更時立即更新
  - `pending`：待執行
  - `in_progress`：執行中（同時只能有一個）
  - `completed`：已完成
- **完成確認**：任務完成後必須標記為 completed，不得批次更新
- **進度追蹤**：複雜專案必須定期回報 TodoList 進度

### 互動規範
- **需求澄清優先**：收到需求時，優先澄清需求而非立即開始開發
- **計劃先行**：複雜任務必須先制定計劃，獲得確認後再執行
- **過程透明**：開發過程中的重要決策需要說明原因
- **錯誤回報**：遇到問題時詳細描述問題現象與可能原因

## 9. 程式碼規範 (依照共同規範)

### 模組化原則
- **功能分離**：一個功能一支程式，禁止多功能耦合在單檔內
- **檔案註解**：每支程式碼檔案需包含檔頭註解
  - 功能說明：清楚描述程式主要功能和用途
  - 建立日期：檔案創建日期
  - 版本號：當前版本資訊
  - **編碼聲明**：程式檔案頂端必須加入「請以繁體中文產生程式碼註解。請務必保持 UTF-8 編碼。」說明
- **程式規模**：單支程式碼不得超過 500 行，超過需拆分為 core 與 utils
- **擴展機制**：新增功能必須建立新程式檔案，不得直接堆疊於舊檔案中
- **錯誤處理**：需完整處理錯誤，並清楚回報錯誤來源模組

### 代碼修改規範
- **補丁優先**：所有代碼修改必須使用補丁方式（patch-based modification），嚴禁使用 shell 指令直接修改程式檔案
- **修改工具**：僅允許使用 Edit、MultiEdit、Write 等專用工具進行代碼修改
- **變更追蹤**：每次修改必須能清楚追蹤變更內容，確保修改過程的可控性與可回溯性
- **安全考量**：避免使用 sed、awk、perl 等 shell 文字處理工具修改程式碼，防止編碼問題與意外變更

## 10. AI 計畫記錄規範 (依照共同規範)

### 計畫記錄機制
- **強制記錄**：AI 收到符合條件的指令後，必須將執行計畫記錄到專案目錄的計畫記錄區
- **記錄內容**：包含收到的指令/需求與制定的執行計畫步驟清單
- **執行追蹤**：使用 checkbox 格式追蹤每個步驟的完成狀態
- **延續性**：確保新視窗開啟後能延續之前的計畫與進度
- **詳細化記錄**：記錄足夠詳細的資訊，讓新的 AI 實例能完全理解並繼續執行

### 計畫檔案同步規範（v2.3 新增 2026-02-10）
- **專案歸檔**：每次使用 Claude Code 計畫模式（Plan Mode）完成計畫後，必須將 `~/.claude/plans/` 中新產生的計畫檔案移動到當前專案的 `.claude/plan/` 目錄
- **目錄規範**：每個專案必須有 `.claude/plan/` 資料夾，用於存放該專案的計畫檔案
- **執行時機**：計畫完成並獲得使用者確認後，立即執行檔案搬移
- **搬移方式**：使用移動（Move）而非複製，避免全域目錄累積過多檔案
- **檔案識別**：根據計畫內容中的專案路徑、技術棧、功能描述判斷所屬專案
- **注意事項**：`~/.claude/plans/` 為 Claude Code 系統預設儲存位置，無法透過設定變更，因此需手動搬移

### 需要建立計畫的指令類型
- **功能開發**：新增功能、模組開發、元件建立等
- **錯誤修復**：Bug 修正、問題排除、系統修復等
- **程式碼重構**：模組化、架構調整、程式碼優化等
- **效能優化**：系統效能改善、資料庫優化等
- **使用者介面調整**：UI/UX 改進、界面修改等
- **資料處理**：資料遷移、格式轉換、資料清理等
- **整合功能**：API 整合、第三方服務整合等
- **測試相關**：測試撰寫、測試環境建立等

### 不需要建立計畫的指令類型
- **文檔維護**：更新架構圖.md、修改文檔內容等
- **版本控制**：上傳 git、提交變更、推送到遠端倉庫等
- **檔案管理**：檔案移動、重新命名、檔案清理等
- **環境配置**：設定檔調整、環境變數配置等
- **簡單查詢**：檔案內容查看、狀態檢查等
- **單純資訊提供**：回答問題、說明文檔、技術解釋等

### 目錄結構規範
- **計畫記錄區**：專案根目錄中建立「計畫記錄區」資料夾
- **計畫檔案**：在計畫記錄區中建立英文檔名的計畫檔案
- **命名規則**：`功能類型-具體名稱-plan.md` 格式
  - 例如：`implement-user-auth-plan.md`、`fix-database-connection-plan.md`
  - 功能類型：implement、fix、refactor、optimize、update 等
  - 具體名稱：使用英文短橫線分隔，描述具體內容
- **檔案管理**：每個計畫一個獨立的 md 檔案，便於追蹤與管理
- **專案位置標記**：每個計畫檔案必須明確標記所屬專案的絕對路徑

### 記錄格式規範
```markdown
# 計畫名稱（繁體中文）

## 專案位置
**絕對路徑**：`C:\Users\[username]\[project_path]`
**相對根目錄**：`./[relative_path]`

## 計畫概述
```

## 11. Git 與版本控管

### 分支管理策略（2026-03-20 建立）

#### 分支結構
- **main**：穩定版本，僅在發版時從 develop 合併，未來直接對應正式部署環境
- **develop**：日常開發主線，所有 commit 預設在此分支進行
- **feature/***：大型功能才從 develop 切出，完成後合回 develop 並刪除
- **hotfix/***：（上線後啟用）正式環境緊急 bug 修復，從 main 切出，修完合回 main + develop

#### 日常開發規則
- **預設分支**：所有日常開發 commit 到 `develop` 分支，禁止直接 commit 到 `main`
- **commit 前確認**：執行 commit 前必須先確認目前在 `develop` 分支（`git branch --show-current`）
- **發版流程**：develop 穩定後 → 切到 main → `git merge develop` → 打 Tag（`vX.Y.Z`）→ 切回 develop
- **feature 分支**：僅在大型功能（預計影響 10+ 檔案或開發超過 3 天）時才從 develop 切出
- **feature 命名**：`feature/功能英文簡述`（例：`feature/profile-analysis`、`feature/proposal-export`）

#### 禁止事項
- 禁止直接在 main 上開發或 commit（發版合併除外）
- 禁止 force push 到 main 或 develop
- 禁止刪除 main 或 develop 分支

### Commit Message 規範
**格式**：`[類型] 模組: 簡述`

**範例**：
- `[功能] login: 新增多因素驗證流程`
- `[修復] api: 修正錯誤回傳格式`

### Pull Request 準則
- PR 必須對應 `CLAUDE.md` 的更新
- 未附帶文件更新的 PR 一律不得進入主分支
- **版本標記**：每次正式釋出須建立 Tag（遵循 `vX.Y.Z` 格式）

## 12. 安全與配置

- **禁止硬編碼**：
  - 所有可配置的值（URL、端口、路徑、閾值、功能開關等）一律禁止硬編碼於程式中
  - 必須使用環境變數（`.env`）或設定檔進行管理
  - 前端使用 `import.meta.env.VITE_*` 讀取環境變數
  - 後端使用 `os.getenv()` 或 Pydantic Settings 讀取環境變數
  - 常數值若需跨模組共用，應定義於專屬的 constants 或 config 模組中
  - 新增環境變數時，必須同步更新 `.env.example` 範本檔案
- **敏感資訊管理**：
  - 禁止將密碼、API Key 等敏感資訊硬編碼於程式中
  - 統一存放於 `.env` 或安全金鑰管理服務
- **前端敏感資料儲存加密**：
  - 所有存放於瀏覽器端（localStorage、sessionStorage）的敏感資料（認證狀態、Token、使用者資訊等）必須加密後再儲存，嚴禁以明文形式存放
  - 必須建立獨立的 `secureStorage` 工具模組，提供 `setSecureItem`、`getSecureItem`、`removeSecureItem` 等加密存取介面
  - 加密金鑰必須從環境變數讀取（前端使用 `import.meta.env.VITE_STORAGE_ENCRYPTION_KEY`），禁止硬編碼於程式中
  - 加密機制須包含隨機 IV（初始化向量），確保相同明文產生不同密文
  - 解密失敗時（金鑰變更或資料損壞）須自動清除無效資料，不得導致應用程式崩潰
  - 目的：防止瀏覽器 DevTools 直接讀取明文、防止瀏覽器擴充套件掃描竊取、防止肩窺攻擊
- **輸入驗證**：所有外部輸入需經過檢查與清理，避免 XSS / SQL Injection 等攻擊
- **國際化準備**：程式須預留 i18n 架構，以利未來國際化擴充
- **資料庫欄位自動遷移**：
  - 所有新增的資料庫欄位，都必須實作「重啟後自動新增」的遷移邏輯
  - 在對應管理器的 `__init__()` 中呼叫 `_ensure_xxx_column()` 方法
  - 使用 `INFORMATION_SCHEMA.COLUMNS` 檢查欄位是否存在，不存在時執行 `ALTER TABLE ADD`
  - 必須用 `try/except` 包裝，遷移失敗不得阻擋後端啟動（僅 `logger.warning`）
  - 新增欄位應設定合理的 `DEFAULT` 值，避免 `NULL` 導致查詢異常
  - 每次新增欄位遷移後，必須在對應模組的 `CLAUDE.md`「自動遷移清單」中登記

## 15.1 AI 系統操作整合規範（v3.1 新增 2026-04-27）

> 本規範適用於所有含 AI 對話助手的專案。核心理念：**系統的每個功能，AI 都應該能操作**。

### AI ACTION 架構

系統中的 AI 對話助手可透過 `<<<ACTION>>>` 標記直接操作系統功能。新增任何使用者可操作的功能時，必須同步考慮 AI ACTION 整合。

### 新增系統功能時的 AI 整合 Checklist

每次新增使用者可操作的功能（按鈕、開關、CRUD、設定變更等），必須評估以下項目：

1. **能力定義**：在 `prompts/chat/proposal_chat_actions/` 中新增對應的動作定義
   - 動作名稱、用途說明、情境描述、參數定義、範例
   - **禁止使用關鍵字匹配**，AI 必須靠語意理解判斷使用者意圖
   - 不確定時 AI 應反問使用者

2. **後端 Handler**：在 `proposals_chat_actions.py` 中新增執行函式
   - 函式命名：`_execute_{action_name}(action: dict, proposal_id: str) -> dict`
   - **必須回傳 `previousState`**：記錄修改前的狀態，供前端還原使用
   - 回傳格式：`{"success": True, "type": "action_name", "previousState": {...}}`

3. **還原（Undo）支援**：在 `undo-action` 端點中新增反向操作邏輯
   - 每個可修改資料的 ACTION 都必須支援還原
   - 查詢型 ACTION（不改資料）不需要還原

4. **前端事件處理**：在 `ProposalWorkspace.tsx` 的 `ai-action-executed` 監聽器中新增對應的 UI 刷新邏輯
   - 結構性變更 → 重新載入建議書
   - block 層級變更 → triggerExternalUpdate
   - 特殊操作（如匯出）→ 觸發對應的前端行為

5. **Context 注入**：確保 AI 有足夠的背景資料來正確執行操作
   - 在 `proposal_chat_context_builder.py` 中注入相關的 ID 和狀態
   - AI 需要知道 ID 才能填 ACTION 參數

### 設計原則

- **AI 靠語意理解判斷**：不在 Prompt 中寫「當使用者說XX時執行YY」這種規則
- **每個操作都能還原**：使用者按「還原」可以回復 AI 的任何操作
- **不確定就反問**：AI 不確定使用者意圖時，先反問而非擅自操作
- **操作後即時刷新**：前端透過事件機制即時反映操作結果
- **Handler 獨立模組**：ACTION handler 獨立為 `proposals_chat_actions.py`，不汙染路由檔

## 16. 部署與交付規範

> 本章為強制性規範，適用於所有需要部署至測試/正式環境的專案。

### 測試環境還原標準 SOP

| 步驟 | 執行者 | 動作說明 |
|------|--------|----------|
| 01 | MIS / DevOps / Infra | 測試開始前，對 VM 執行快照（Snapshot），確保作業系統與 Docker 環境均被涵蓋 |
| 02 | 開發者 | 部署容器服務：執行 `docker-compose up`，確認服務健康 |
| 03 | QA / 弱掃 | 執行弱點掃描或功能測試，若發現問題記錄 Issue |
| 04 | 開發者 | 依據 Issue 修正程式碼，重新 Build Image，更新部署 |
| 05 | MIS / DevOps / Infra | 測試結束後若需還原，執行 VM 快照回滾（Rollback） |
| 06 | 開發者 | 重新執行 `docker-compose up` 啟動服務，完全不需手動重建資料庫 |

### HTTPS 測試環境一致性要求

- **環境一致原則**：測試環境必須與正式環境一致，開通 HTTPS 443 並掛載 SSL 憑證
- **禁止僅用 HTTP 測試**：以 HTTP 8080 測試的弱掃結果不具合規性參考價值
- **避免雙設定維護**：測試與正式環境須使用相同的協定設定，避免開發者維護兩套設定
- **Nginx 前置要求**：若測試時是開 HTTP 8080，必須確保已在前方架設 Nginx 處理 HTTPS

### 開發者交付清單

每次部署至測試/正式環境，開發者**必須**準備以下交付物：

| 交付物 | 說明 |
|--------|------|
| **Docker Image** | 已通過本地測試的映像檔，附上版本標籤（Tag） |
| **docker-compose.yml** | 定義服務拓撲、資料庫版本、Port 對應、環境變數 |
| **.env.example** | 告知 MIS 需填入哪些連線字串、API Key、密碼等 |
| **Init SQL 腳本** | 放置於容器啟動時自動執行的資料夾，確保資料庫表格自動初始化 |
| **README.md** | 簡易啟動說明，含健康檢查指令（`docker ps` / `curl`） |

MIS 團隊**必須**準備以下環境與配置：

| 交付物 | 說明 |
|--------|------|

| **網路配置** | 防火牆開放 80 / 443 Port；設定 Reverse Proxy（Nginx）將流量導向容器 |
| **SSL 憑證** | 負責申請、更新與掛載 SSL 憑證，確保 HTTPS 443 可正常存取 |

#### 階段六：Production（Owner: MIS，RD 負責交付與驗證）

**RD 工作流程：**
1. 合併 MR 至 main（附：Stage 結果、滲透測試報告、Rollback 計畫）
2. 建立 Release Tag（`vX.Y.Z`），版本號由 RD 提供
3. Tag 觸發 CI 自動 Re-build + Re-scan
4. 交付部署物給 MIS
5. 部署後功能驗證 30 分鐘

**RD 交付給 MIS：**

| 交付物 | 說明 |
|--------|------|
| Image tag | 正式版標籤（`:v1.2.3`） |
| docker-compose.yml | 部署配置檔 |
| 環境變數清單 | `.env.example` 已更新 |
| DB Migration Script | migrate up + migrate down 成對 |
| Rollback 計畫 | 回滾步驟 + 判斷條件 + DB 回滾 Script |

**DB Migration 回滾要求：**
- Migration 必須可逆（migrate up + migrate down 成對交付）
- 不可逆操作（如 `DROP COLUMN`）須在部署計畫中明確標註
- Rollback 計畫中須包含 DB 回滾 Script，供 MIS 緊急執行
- Claude 撰寫 Migration 時，必須同時產生對應的 Down Migration

**Rollback 觸發條件（任一成立即啟動）：**
- Health Check 持續失敗超過 3 分鐘
- Error Rate 超過基線 5 倍
- P0/P1 使用者回報問題
- 部署後 30 分鐘內監控異常

**Hotfix 流程：**
1. 從 main 建 `hotfix/*` 分支 -> 修復 + 測試
2. MR to main（`[HOTFIX]` 標記，1 人快速審核）-> 建 Patch Tag（`v1.2.4`）
3. CI 自動建置 -> RD 交付部署物給 MIS -> MIS 部署 + Health Check
4. Cherry-pick 回 develop + staging -> 24hr 內完成 Post-mortem
- Security Scan 與 Health Check 不可跳過，Code Review 可簡化，UAT 可事後補簽

### Claude 自動協助規則

> Claude 在開發過程中應主動協助以下工作：

- **單元測試**：撰寫程式碼時，主動產生對應的單元測試
- **CHANGELOG**：功能完成時，主動產生 CHANGELOG 條目
- **Dockerfile**：協助撰寫符合打包標準的 Dockerfile
- **DB Migration**：撰寫 Migration 時，必須同時產生 migrate up 與 migrate down
- **環境變數**：新增環境變數時，主動提醒更新 `.env.example`
- **Rollback 計畫**：發版前協助撰寫 Rollback 計畫，含 DB 回滾 Script
- **Staging 驗收**：可依據 CHANGELOG 自動產生測試案例
- **弱點修復**：修復安全弱點時，確保不引入新問題，重跑相關測試
