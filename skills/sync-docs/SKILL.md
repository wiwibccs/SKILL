# 專案文件同步更新工具

你是專案文件的同步更新助手。根據指令執行不同模式。

## 模式判斷邏輯

1. 如果參數為 `changelog` → 執行【僅更新 CHANGELOG】
2. 如果沒有參數 → 執行【完整同步】（架構圖 + API 契約 + CHANGELOG）

---

## 【完整同步】

依序執行以下三個階段：

### 階段一：更新架構依賴圖

#### 1.1 掃描專案結構

掃描整個專案的檔案結構，收集以下資訊：

```
前端掃描（如果存在 frontend/ 或 src/）：
- 頁面元件結構（pages/ 或 components/）
- 狀態管理（stores/）
- 服務層（services/）
- 型別定義（types/）
- 工具函數（utils/）
- Hooks（hooks/）
- Context（contexts/）
- 每個檔案的用途（從檔頭註解或檔名推斷）
- 每個檔案的行數

後端掃描（如果存在 backend/ 或 app/）：
- API 路由結構（api/）
- 核心服務（core/）
- Schema 定義（schemas/）
- 資料庫模組（database/）
- 排程模組（scheduler/）
- 每個檔案的用途（從檔頭註解或 docstring 推斷）
- 每個檔案的行數
```

#### 1.2 更新根目錄 CLAUDE.md

找到專案根目錄的 `CLAUDE.md`，定位到 `## 架構依賴圖` 區塊，用掃描結果重新生成架構圖。

**更新規則**：
- 保持既有的樹狀圖格式（使用 `├─` `└─` `│` 符號）
- 每個檔案/模組標註用途說明（括號內簡述）
- 標註行數（超過 200 行的檔案）
- 標註版本號（如果檔頭註解有版本資訊）
- 新增的檔案/模組要加入樹狀圖
- 已刪除的檔案/模組要從樹狀圖移除
- 已改名的檔案/模組要更新名稱
- **不修改架構依賴圖以外的區塊**

**格式範例**：
```
├─ moduleName/（模組用途說明）
│   ├─ fileName.tsx（功能說明 - 補充細節，XXX行）
│   └─ utils.ts（工具函數）
```

#### 1.3 更新子模組 CLAUDE.md

找到所有子模組資料夾中的 `CLAUDE.md`，逐一更新其 `## 架構依賴圖` 區塊。

**掃描範圍**：
```
用 glob 搜尋所有 CLAUDE.md：
- frontend/**/CLAUDE.md
- backend/**/CLAUDE.md
- 其他子模組路徑下的 CLAUDE.md
```

**更新規則（同根目錄）**：
- 只更新該子模組自己的架構依賴圖，不動其他區塊
- 如果子模組下有新增/刪除/改名的檔案，同步更新
- 如果子模組的 CLAUDE.md 沒有架構依賴圖區塊，新增該區塊

#### 1.4 架構圖更新摘要

完成後記錄變更摘要，供後續寫入 CHANGELOG 使用：

```
架構圖更新摘要：
- 根目錄 CLAUDE.md：新增 X 個、移除 X 個、更新 X 個節點
- [子模組路徑]/CLAUDE.md：新增 X 個、移除 X 個、更新 X 個節點
- ...
```

---

### 階段二：更新 API 契約

#### 2.1 掃描 API 端點

掃描後端 API 路由，收集所有端點資訊：

```
掃描來源（依優先順序）：
1. backend/app/api/ 目錄下所有 .py 檔案
2. 找 @app.get / @app.post / @app.put / @app.delete / @app.patch / @router 裝飾器
3. 從 Pydantic Schema（backend/app/schemas/）提取請求/回應模型
4. 從路由檔案中的 docstring 提取端點說明

收集欄位：
- HTTP 方法（GET/POST/PUT/DELETE/PATCH）
- 路徑（/api/xxx）
- 說明（從 docstring 或註解提取）
- 請求模型（從 Schema 提取）
- 回應模型（從 Schema 提取）
- 所屬模組/路由群組
```

#### 2.2 更新 API_CONTRACT.md

找到專案根目錄的 `API_CONTRACT.md`，用掃描結果更新。

**更新規則**：
- 按模組/功能分組排列端點
- 每個端點標註 HTTP 方法、路徑、說明
- 新增的端點加入文件
- 已刪除的端點從文件移除
- 已變更的端點更新描述
- 保持既有的文件格式與風格
- 如果 API_CONTRACT.md 不存在，提示使用者是否要建立

#### 2.3 API 契約更新摘要

```
API 契約更新摘要：
- 新增端點：X 個
- 移除端點：X 個
- 更新端點：X 個
```

## 【僅更新 CHANGELOG】

### 步驟 1：收集變更資訊

```
資訊來源：
1. git log --oneline（從上次 CHANGELOG 記錄的版本到現在的所有 commit）
2. git diff --stat（如果有未提交的變更）
3. 如果是完整同步模式，使用前兩個階段的變更摘要
```

### 步驟 2：判斷版本號

從 CLAUDE.md 的 `**文檔版本**` 或 CHANGELOG.md 的最新記錄取得目前版本號。

**版本號規則**：
- 讀取專案既有的版本號格式（例如 `v1.78.622`）
- 如果是 `vX.Y.Z` 格式，Z 遞增 1
- 如果無法判斷，詢問使用者

### 步驟 3：寫入 CHANGELOG.md

在 CHANGELOG.md 的最上方（標題下方）插入新的變更記錄。

**格式**：遵循專案既有的 CHANGELOG 格式。如果沒有既有格式，使用以下預設格式：

```markdown
## [vX.Y.Z] - YYYY-MM-DD

### 變更內容
- [類型] 模組: 變更說明

### 文件更新
- 更新 CLAUDE.md 架構依賴圖
- 更新 API_CONTRACT.md
```

**類型標籤**：
- `[功能]`：新增功能
- `[修復]`：錯誤修復
- `[重構]`：程式碼重構
- `[文件]`：文件更新
- `[優化]`：效能或體驗優化

### 步驟 4：同步更新 CLAUDE.md 版本號

更新專案根目錄 CLAUDE.md 底部的 `**文檔版本**` 和 `**最後更新**`。

## 完整同步的輸出摘要

```
========================================
  文件同步更新完成
========================================
更新日期：YYYY-MM-DD
專案版本：vX.Y.Z → vX.Y.Z+1

架構依賴圖：
  根目錄 CLAUDE.md ............ 更新（+X -X ~X 節點）
  [子模組1]/CLAUDE.md ......... 更新（+X -X ~X 節點）
  [子模組2]/CLAUDE.md ......... 無變更
  ...

API 契約：
  API_CONTRACT.md ............. 更新（+X -X ~X 端點）

CHANGELOG：
  CHANGELOG.md ................ 新增 vX.Y.Z 記錄

已更新檔案清單：
  1. CLAUDE.md（架構依賴圖 + 版本號）
  2. [子模組]/CLAUDE.md
  3. API_CONTRACT.md
  4. CHANGELOG.md
========================================
```

---

## 注意事項

- 所有輸出使用繁體中文
- 只更新指定的區塊（架構依賴圖、API 端點），不動其他內容
- 更新前先讀取原始內容，確保不覆蓋人工編輯的部分
- 如果掃描結果與現有內容完全一致（無變更），跳過更新並回報「無變更」
- 如果某個文件不存在（例如沒有 API_CONTRACT.md），詢問使用者是否要建立
- 架構圖中的版本標註（例如 `v1.78.546 新增`）保留不動，除非該模組已被刪除
- CHANGELOG 的既有記錄不可修改，只能在最上方新增
